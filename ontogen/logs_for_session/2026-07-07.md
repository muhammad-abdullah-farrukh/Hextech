# Session: 2026-07-07

## What was wrong
- llama-server (llama.cpp) was falling back to CPU inference (~2.9 tok/s) instead of GPU
- Root cause: build was pinned to CUDA 13.3 toolkit, but installed driver (575.64.03) only supports up to CUDA 12.9 — binary compiled but couldn't initialize CUDA at runtime
- Secondary issue: after switching nvcc to 12.4, CMake was still resolving the CUDA toolkit path via the /usr/local/cuda symlink, which pointed to 13.3 — pulled in compute_120 (Blackwell) architecture that 12.4's nvcc doesn't support, causing a hard compile failure

## What we fixed
- Rebuilt llama.cpp pinning both the compiler AND toolkit root to CUDA 12.4:
  cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_COMPILER=/usr/local/cuda-12.4/bin/nvcc -DCUDAToolkit_ROOT=/usr/local/cuda-12.4 -DCMAKE_CUDA_ARCHITECTURES=89 -DGGML_CUDA_COMPRESSION_MODE=none
- Restricted CMAKE_CUDA_ARCHITECTURES to 89 (Ada only) instead of the default 7-arch list, avoiding unsupported architectures and speeding up compilation
- Build completed clean at 100%

- First rebuild attempt only pinned the compiler, not the toolkit root — CMake still found CUDAToolkit 13.3 via symlink and generated an incompatible architecture list, causing nvcc fatal: Unsupported gpu architecture 'compute_120'

# Session 2: 2026-07-07 — pipeline resumability + graph correctness (duplicate nodes, hallucinated entities)

## What was wrong

1. **No way to resume `pipeline.py` from a specific stage.** All 9/10 stages'
   intermediate output only lived in Postgres (`pipeline_runs`,
   `graph_entities`/`graph_relationships`) — nothing on disk, so there was no
   way to inspect or hand-edit an intermediate artifact (e.g. the generated
   ontology) and re-run just the downstream stages without a full re-run.

2. **Duplicate graph nodes for the same real-world entity** — e.g.
   `scikit-learn` existed as two separate Neo4j nodes,
   `.../entity/scikit_learn` and `.../entity/Scikit_learn`, splitting its
   edges across two identities.

3. **Hallucinated entity substitution (severe).** Résumé education entries
   were replaced with completely wrong institutions in the graph:
   `"Air University"` → `"Brown University"`, `"Pakistan Intl. School"` (a
   Matric-level high school) → `"Birla Institute of Technology and Science,
   Pilani"` (an Indian university) — confirmed the corruption was introduced
   in Ontogen, not Phase 1 parsing (`04_structured.json` had the correct
   institution names).

## What we fixed

**1. Disk mirroring for every stage** (`ontogen/pipeline.py`):
   `outputs/stages/<resume_uuid>/stageN_*.json|.ttl` now gets written
   alongside every existing Postgres write (`_save_stage_to_disk`), purely
   additive — nothing reads these back on its own.

**2. `--from-stage=<stage>` resume mode** (`ontogen/pipeline.py`): loads every
   stage strictly before the given one from disk instead of recomputing it
   (`_STAGE_ORDER`, `_skip_before`, `_stage_value`, `_load_stage_from_disk`).
   Added a new disk-only artifact, `stage3_5_edc_canon_result.json` (the
   post-EDC `match_results`), since nothing previously persisted that
   intermediate value — needed to resume cleanly right before the `ontology`
   stage. Valid stage keys: `cq_gen, cq_answer, relation_extract,
   match_validate, edc_canon, ontology, stage9_10`. Must be single-token
   (`--from-stage=x`, not two args) — the CLI parser has no `argparse` and
   would otherwise swallow the value as the résumé UUID.

**3. Duplicate-node root cause #1 — `_slugify()` didn't case-fold**
   (`ontogen/db/kg_staging.py`): `"Scikit-learn"` and `"scikit-learn"` slugged
   to different URIs (`Scikit_learn` vs `scikit_learn`), so Neo4j's
   `MERGE (e:Entity {uri: ...})` treated them as different nodes. Fixed by
   lowercasing before slugifying — the one choke point every non-QID entity
   URI passes through.

**4. Duplicate-node root cause #2 — gazetteer lookups scoped too strictly by
   `entity_type`** (`ontogen/db/gazetteers.py`, `ontogen/stages/
   canonicalize.py`): the gazetteer only had `"skill"`-typed rows for
   `scikit-learn`, none `"technology"`-typed, so a mention classified as
   `"technology"` missed Tier 1/2 entirely and fell to Tier 3's free-form LLM
   generation (inconsistent casing). Added `lookup_any_type`,
   `canonical_values_any_type`, `get_qid_any_type` as fallbacks used by
   `_tier1`/`_tier2`/`_get_qid` when the type-scoped query misses.

**5. Hallucinated-entity root cause — Tier 2 (embedding) blindly accepted a
   cosine-similarity match ≥0.88 with no verification step**
   (`ontogen/stages/canonicalize.py`). Confirmed empirically against the live
   resolver + full gazetteer + production embedding model
   (`BAAI/bge-small-en`): `"Air University"` scored **0.884** against
   `"Brown University"` — just over the threshold — because generic shared
   words ("University") dominate short-string embeddings for entities with
   little other semantic content. This is the *entity* analogue of a bug
   `stage6_match_validate.py` already had to fix for *relationship* matching
   (see its `AUDIT NOTE` — same failure shape: embedding retrieves a
   plausible-but-wrong candidate, nothing verifies it before accepting).
   Fixed by mirroring stage6's retrieve-then-verify design:
   - `_content_tokens` / `_is_acronym_expansion` / `_passes_lexical_guard`: a
     deterministic guard rejecting a "match" that shares zero distinguishing
     words with the mention (exempting recognized acronym expansions like
     `MIT` → `Massachusetts Institute of Technology`).
   - `_llm_confirms_entity_match` (new): Tier 2 now requires an LLM to
     confirm a same-string/non-acronym candidate before accepting it —
     `_ENTITY_VERIFY_PROMPT`, `_ENTITY_ACCEPT_CONFIDENCE_THRESHOLD = 0.65`,
     same "yes 87"/"no 12" format stage6 already uses. Only fires for the one
     candidate that already cleared the cosine threshold — not per-entity.
   - Tier 3's own prompt (`_ENTITY_LLM_PROMPT`) rewritten to explicitly
     forbid substituting a different entity ("a wrong answer is worse than no
     answer"), instead of the old "normalize to most widely recognized form"
     phrasing that invited exactly this failure.
   - `_llm_confirms_entity_match`'s `call_llm` bumped to `max_tokens=600`
     (was 200) — this reasoning model burns unpredictable budget on
     `<think>` reasoning before the real answer; 200 was truncating to empty
     content, which happened to fail *safe* (empty → parsed as "no" → falls
     through to Tier 3) but silently defeated Tier 2 on every borderline
     match.

## What went wrong (during the fix)

- **First diagnosis of the hallucination was wrong.** Initially assumed Tier
  3 (LLM) was the culprit based on its "normalize to most widely recognized
  form" prompt looking suspicious, and shipped a prompt rewrite + lexical
  guard for Tier 3 alone. Live-testing against the real resolver afterward
  showed the fix didn't work — the entity never reached Tier 3 at all,
  because Tier 2 was accepting it first. An earlier manual embedding test
  that seemed to rule out Tier 2 (`0.71` similarity, safely under 0.88) used
  a smaller hand-picked candidate list and a slightly different embedding
  model name (`bge-small-en-v1.5` vs. the production `bge-small-en`) — against
  the real 147-entry gazetteer and the real model, the score was `0.88+`.
  Lesson: when root-causing a resolver/matching bug, reproduce against the
  *exact* production model name and the *full* real candidate set, not a
  representative subset — a "safely below threshold" result on a partial
  repro is not evidence the real path is safe.
- Repeated pipeline re-runs during unrelated debugging (checking whether the
  process was "stuck") accumulated duplicate `graph_entities`/
  `graph_relationships` rows for the same résumé, since `kg_staging.py`'s
  writers are plain `INSERT`s with no upsert/dedup — had to be manually
  cleared (`DELETE ... WHERE source_doc = ...`) before re-staging.
- `load_to_neo4j.py --wipe` clears Neo4j but does **not** reset Postgres's
  `synced_to_neo4j` flags — after a manual `cypher-shell ... DETACH DELETE`
  wipe, reloading only pushes rows still marked unsynced, silently dropping
  any résumé whose rows were already marked synced before the wipe. Had to
  scope cleanup to one résumé at a time (`DELETE ... WHERE source_doc = ...`
  in Postgres + a targeted `DETACH DELETE` of just the known-bad Neo4j nodes)
  rather than a full wipe, to avoid losing the other two résumés' correct
  data.

## Follow-ups for CLAUDE.md
- `outputs/stages/<resume_uuid>/` now mirrors every pipeline stage to disk;
  `--from-stage=<stage>` (single-token form only) resumes from it.
- `kg_staging.py` writers are plain INSERTs, no dedup — re-running the
  pipeline for an already-processed résumé silently duplicates its
  `graph_entities`/`graph_relationships` rows rather than upserting them.
- `load_to_neo4j.py --wipe` does not reset `graph_entities`/
  `graph_relationships.synced_to_neo4j` in Postgres — after a manual Neo4j
  wipe, resyncing only pushes rows still marked `FALSE`, silently omitting
  already-synced résumés unless their flags are reset too.
- Entity resolution (`ResumeEntityResolver` in `canonicalize.py`) has 3
  tiers: gazetteer exact-match, gazetteer embedding match (Tier 2, now
  LLM-verified before accepting), LLM free-form normalization (Tier 3).
  Wikidata's own embedding space (`wikidata_properties`) is used only for
  *relationship*/predicate matching in `stage6_match_validate.py`, never for
  entities — entity Tier 2 matches against the local gazetteer's own
  canonical values.
- No `psql` client on the host — use `sudo docker exec
  ocr-resume-paser-postgres-1 psql -U resume_parser -d resume_parser -c
  "..."` for any Postgres query. Neo4j: `cypher-shell -a
  bolt://127.0.0.1:7687 -u neo4j -p 5053811238 "..."`.
- When root-causing a resolver/embedding-matching bug, always reproduce
  against the exact production model name and the full real candidate
  set/gazetteer — a partial repro that looks "safely below threshold" is not
  reliable evidence.
